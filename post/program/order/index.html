订单业务的初步理解

商品表

  1. 字段设计bit与表示这个商品是否在限制某种用户限制, 用户总库存, 用户年库存, 用户月库存, 用户周库存, 用户日库存

订单商品表

  1. 字段设计订单id_商品id做联合主键

订单创建接口

1. 客户端点击创建订单
   用户ID
   商品ID
   门店ID(业务需要)

2. 服务端
   1. 参数校验
      _sid在中间件校验了用户合法性

   2. 判断商品表是否存在某种用户限制
   不存在限制:
      1. 写入rocketmq事物消息记录订单所有信息, 2. lua脚本: 记录订单扣减流水和扣减总库存, 3. 创建订单信息和订单商品表(mysql事物) commit, 4. 上述流程的处理流程全都正确, commit rocket事物消息, 否则按照下面的逻辑处理
             1.失败直接返回客户端rocket错误,服务器繁忙
             2.失败库存不足.commit事物不需要要补偿了
             3.失败返回创建订单失败,服务器繁忙, 不要commit rocket事物,需要补偿最终一致性
             4.事物消息本身的commit成功失败不需要处理, 因为后续的事物的补偿是业务幂等的多执行几次或者不执行都不影响业务正常运行
         消息队列事物补偿逻辑:

   存在限制:
      1. 先进行用户限制的简单判断,不满足直接返回给用户库存不足或者达到限制条件, 主要是为了用户体验
      2. 和不存在限制的流程一样, 只是需要注意以下几点.
         1. lua脚本: 需要加上判断用户限制条件是否满足,不满足 然后记录订单流水, 扣减总库存, 递增用户商品库存限制值.返回的信息也可以根据lua返回值判断是否总库存不足还是达到限制
         2. 消息事物补偿库存lua脚本里还需要补偿递增的用户限制条件值
   消息事物补偿:
     从消息中取出订单信息, 订单是否入库,
                         入库,就不需要管了,代表创建逻辑走完了
                         没有入库, 就回滚库存(如果商品有用户条件限制,那么用户条件限制也需要回滚), 可能创建出现故障了,客户端也不知道订单是创建好了所以直接回滚库存,不要补偿订单了.
   订单自动超时脚本:
      定时执行sql如下: UPDATE status = 自动取消 WHERE status = 待支付 AND 数据库中的三方支付超时字段 < 当前时间 - 预期支付回调时长(30s),多30s的缓冲时间
      定时执行时间取30s, 保证支付超时不超时60内就会取消
      订单状态字段: 1:待支付 2:自动取消 3:已支付 20:待接单 21:已接单 22:配送中 22:配送异常  23:配送完成 100:完成 111:已退款
      后续就算是三方支付接口: 我们只需要判断是status <= 自动取消状态 立即 变更回来已支付即可,即可冲正状态

注意点:
1. redis集群模式我猜测lua脚本会有先后性问题, 设计规划的时候确保处于一个key下,这样就保证会分发在一个slot,从而lua能执行是单线程
2. 事物补偿逻辑做幂等性, 事物补偿逻辑因为只有redis和mysql所以只需要判断最后的mysq,即可确定redis如何补偿, 还有其他步骤依然需要细细斟酌补偿机制.

订单创建支付二合一接口

多了一个调用三方支付接口

      1. 写入rocketmq事物消息记录订单所有信息, 2. lua脚本: 记录订单扣减流水和扣减总库存, 3. 创建订单信息和订单商品表(mysql事物)
      4. 调用第三方支付接口                  5. 上述流程的处理流程全都正确, commit rocket事物消息, 否则按照下面的逻辑处理
             1.失败直接返回客户端rocket错误,服务器繁忙
             2.失败库存不足.commit事物不需要要补偿了
             3.失败返回创建订单失败,服务器繁忙, 不要commit rocket事物,需要补偿最终一致性
             4.失败了直接返回创建支付失败给客户端, 且commit rocket事物, 因为rocket补偿只做了redis和mysql.补偿也没有意义
             4.事物消息本身的commit成功失败不需要处理, 因为后续的事物的补偿是业务幂等的多执行几次或者不执行都不影响业务正常运行
